## Согласование — как писать сценарии (D-4467 + уточнения прогона)

**Оракул настройки** — раздел **согласования требований** в [D-4467](https://hq.devprom.ru/pm/core/D-4467?baseline=123) и [D-8645](https://hq.devprom.ru/pm/core/D-8645?baseline=123): состояние **Согласование**, переход с **Создать согласование**, шаблон согласующих.

> **«Поиск продукта» (первый прогон):** в шаблоне процесса **RequirementState** с согласованием **часто не настроен** — это **не** дефект и **не** SKIP ET-PM-00c+. Эталон берём из **документации**, не из живого ЖЦ **R-*** в проекте. Настройку **историй** и **задач** — **ET-PM-00c+**; **требований** — **ET-PM-27+** (п. 1–2), если в **Состояния требований** перехода ещё нет.

| Слой | Где в UI | ET-сценарий |
|------|----------|-------------|
| **Оракул (документация)** | D-4467 / D-8645 — согласование **требований** | ET-PM-00c+ п. 1 |
| **Проверка R-* в процессе** | **Состояния требований** — может быть **пусто** в «Поиск продукта» | Зафиксировать в **Факт**; настройка — ET-PM-27+ |
| **Шаблон** | `.setup` → **Шаблоны согласований** → **Добавить** | ET-PM-00c+ п. 2 — **история** и **задача**; ET-PM-27+ п. 1 — **требование** |
| **Настройка ЖЦ I-/T-*** | **Состояния историй** (`IssueState`) / **Состояния задач** | ET-PM-00c+ п. 3–4 |
| **Настройка ЖЦ R-*** | **Состояния требований** — по тому же паттерну, что I/T | ET-PM-27+ п. 1–2 *(если не настроено)* |
| **Smoke runtime** | Переход **I-*** / **T-*** → **Согласование** → модуль **Согласование** | ET-PM-00c+ п. 5–7 |
| **Полный runtime I-/T-*** | Завершить лист → состояние артефакта | **ET-PM-27b+** |
| **Полный runtime R-*** | Переход + модуль **Согласование** → завершение | **ET-PM-27+** п. 3–5 |
| **Ручное** | **Согласование** → **Добавить** | **ET-PM-27-b+** |

**Не путать:** состояние ЖЦ **Согласование**, модуль **Согласование** (листы) и **Шаблоны согласований** (настройка состава согласующих).

## Загрузка задачами — как писать сценарии

**Модуль:** иконка **Управление** → sidebar → **Загрузка задачами**. Матрица «участник × календарь»; `#filter-settings` как у списка **нет**.

### Условие попадания задачи / истории на график

| Обязательно | Плюс один из режимов планирования |
|-------------|-----------------------------------|
| **Исполнитель** `[single]` | **Режим A — по трудоёмкости:** **Трудоёмкость** (задача: **Сроки**; история: **Оценка**) |
| | **Режим B — по датам:** **Начать** + **Завершить к** (поля в блоке **Сроки** / на форме истории) |

**Оракулы режимов (после refresh модуля):**

| Режим | Поведение на графике |
|-------|----------------------|
| **A: Исполнитель + Трудоёмкость** | Полоса работы **растягивается** на число **рабочих дней** по **рабочему расписанию** участника (админка **Рабочее расписание** / расписание команды), пока не «уложится» трудоёмкость (с учётом доступности ч/день в проекте — модуль **Участники**) |
| **B: Исполнитель + Начать + Завершить к** | Полоса **от Начать до Завершить к**; **загрузка в каждый день** = **рабочие часы** этого дня по расписанию (равномерное заполнение рабочего времени в интервале) |

**Негатив (оракул ET-PM-25+):** только **Исполнитель** без трудоёмкости и без пары дат — **полосы нет** (или задача не в timeline исполнителя). Только даты без исполнителя — **нет**.

**Сущности:** **T-*** и **I-*** (история на доске) — одинаковое правило; сверять **оба** в ET-PM-25+, если в сценарии есть история.

**Подсказка модуля (trunk):** система также считает **Оценка старта** / **Оценка завершения** по приоритету и очереди работ исполнителя; для ET-оракула достаточно видимой **полосы** и **столбиков загрузки** (оранжевый — недогруз, зелёный — 100%, красный — перегруз).

## Функции — настройка и два рабочих модуля

### Настройка (хаб `.setup`)

| Модуль в UI | Вход | Назначение |
|-------------|------|------------|
| **Уровни функций** | `.setup` → **Уровни функций** (или **Справочники** → типы функций) | Уровни архитектуры; на форме уровня — **Дополнительные настройки** (видимость полей функции) |
| **Состояния функций** | `.setup` → **Состояния функций** (группа **Состояния**) | ЖЦ функций *(в хабе может быть «Состояние функции»)* |

### Рабочие модули (sidebar)

| Модуль в UI | Вход | Представление |
|-------------|------|---------------|
| **Функции** | иконка **Управление** → **Функции** | **Древовидная** структура (иерархия уровней/родитель–потомок); создание — **Активность** |

**Не путать:** **Функции** / **Доска функций** — рабочие модули; **Уровни функций** / **Состояния функций** — только настройка.

**Порядок в ET:** уровни → состояния → **Функции** (дерево, **Активность**) → **Доска функций** (паритет) → привязка **I-*** / **T-***.

### [+] ET-PM-25+ — Загрузка задачами: трудоёмкость, даты, история, отсутствия

**Источник:** S-8552. Документация: D-4699. См. раздел **«Загрузка задачами»** в шапке чартера.

**Предусловия**

- **ET-PM-00d+** выполнен; **N100** — участник проекта **`etpm8549jun23`**.
- *(Рекомендуется)* **ET-PM-00d-c+** — команда с расписанием.
- У **N100** в `.setup` → **Участники** задана **доступность** ч/день (если поле есть).

**Действия**

1. **Негатив — только исполнитель:**
   - **Текущие задачи** → **Задача** «PM: загрузка без плана» → **Приоритет**, **Тип** → **Исполнитель** = N100; **без** **Трудоёмкости** и **без** **Начать**/**Завершить к** → **Сохранить** → **T-neg**.
   - Иконка **Управление** → **Загрузка задачами** → строка **N100** → **T-neg** **не** отображается как полоса загрузки (пустая ячейка / нет в timeline).
2. **Режим A — Исполнитель + Трудоёмкость:**
   - **Задача** «PM: загрузка по трудоёмкости» → **T-loadA**: **Исполнитель** = N100; **Трудоёмкость** = 16ч → **Сохранить**.

3. **Режим B — Исполнитель + Начать + Завершить к:**
   - **Задача** «PM: загрузка по датам» → **T-loadB**: **Исполнитель** = N100; **Начать** и **Завершить к** = интервал **3 рабочих дня** (напр. ближайший пн–ср) → **Сохранить** *(трудоёмкость можно не заполнять)*.
   - **Загрузка задачами** → полоса **T-loadB** от **Начать** до **Завершить к**; в каждый день интервала загрузка ≈ **полные рабочие часы** дня.
4. **История — режим A:**
   - **Доска историй** → **История** «PM: загрузка история» → **I-load**: **Исполнитель** = N100; **Оценка** = 8ч → **Сохранить**.

5. `.setup` → **Участники** → карточка **N100** → **Отсутствие** на 3 дня → **Заместитель** = Hope → **Сохранить**.
6. **Загрузка задачами** — период отсутствия отражён в матрице (снижение загрузки / метка — зафиксировать в **Факт**).

**Ожидаемый результат**

- **Негатив:** **T-neg** **не** на графике при отсутствии трудоёмкости и дат.
- **Режим A:** полоса **T-loadA** / **I-load** по **трудоёмкости** / **оценке** и **рабочему календарю** исполнителя.
- **Режим B:** полоса **T-loadB** по датам; **равномерная** дневная загрузка = рабочие часы каждого дня диапазона.
- Только даты **без** исполнителя или только исполнитель **без** планирования — **нет** на графике.
- Отсутствие и заместитель сохранены на карточке участника и видны в контексте загрузки.

**Факт:** SKIP для **Отсутствие**, если поля нет в процессе; PARTIAL — точный вид отсутствия в матрице на docker.

## Подсказки для исследователя

1. **Роль PM:** сценарии проверяют не «все вкладки ALM», а **рабочий цикл координатора** из [S-8549](https://hq.devprom.ru/pm/core/S-8549).
2. **Формат сценария:** **Предусловия** → **Действия** → **Ожидаемый результат**; построчная сверка с S-855x — в таблице **«Построчное соответствие»** выше.
3. **Документация:** при расхождении UI и ожидания — сверять с [D-4467](https://hq.devprom.ru/pm/core/D-4467?baseline=123); для PM-цикла — **D-4652**, **D-4699**, **D-4712**, **D-4713**, **D-5001** (таблица **«Углубление по документации»** выше).
4. **PARTIAL:** фиксировать поверхность (form / list / board / отчёт) и версию процесса.
5. Глубокие технические цепочки (4 поверхности, custom ЖЦ) — дополнительно `ET-app-deep-chains.md`.
6. **Проект:** новый на прогон; **первый** — «Поиск продукта», одиночный; программа/подпроект — позже.
7. **Участники:** **+ Участник** или массово через **Ещё…** после сброса фильтра.
8. **Согласование:** в **«Поиск продукта»** у **R-*** в процессе **обычно пусто** — оракул **D-4467**; **ET-PM-00c+** — **I-/T-***; **ET-PM-27+** — **R-*** (настройка + цикл); **ET-PM-27b+** — завершение **I-/T-***.
9. **UID:** клик по **названию поля UID** на форме сущности → **Значение по умолчанию** (отдельно от **Вычисляемого поля** в мастере атрибутов).
10. **Custom-атрибут:** мастер **Атрибуты** → **Добавить** — сначала **Сущность** + **Тип атрибута** → **Продолжить**; мат. формула — только при типе **Вычисляемое поле** на шаге 2 карточки.
11. **Итерации:** **Предшественники** на форме итерации + стрелка на графике **План** (не «Дерево работ»).
12. **Зависимости задач:** форма **Предшествующие** + свитчер **`#board-arrows-toggle`** на доске; **строгий запрет ЖЦ** — предикат **«Завершены задачи предшественники»** на переходе **TaskState** (**ET-PM-23+**).
13. **Функции:** настройка — **Уровни** + **Состояния**; работа — **Функции** (дерево, **Активность**) и **Доска функций** (паритет ЖЦ).
14. **Шаблоны:** создание **T-*** / **I-*** — с **доски/списка** (ET-DEEP-PM-00 п. 10–11), не только из `.setup`.
15. **Переход ЖЦ:** пользовательский атрибут (ET-DEEP-PM-00) и системное **Затрачено** (ET-PM-24c+) — разные поля; commit — **Сохранить** в диалоге.
